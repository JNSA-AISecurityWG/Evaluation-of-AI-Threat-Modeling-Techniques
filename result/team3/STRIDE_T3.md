# STRIDE
## 内部にAI機能を有するアプリケーション
### STRIDEテーブル
#### TB1

| TB1  | 緩和策                                                       | 脆弱性                                                       |
| ---- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| S    | 強固な認証（MFA）、セッション管理の厳格化                    | 他の顧客になりすましてログインし、他社の製品画像や評価結果を閲覧する |
| T    | TLSによる通信暗号化、データ完全性チェック                    | ブラウザとフロントエンド間で「Evaluation  results（評価結果）」を改ざんし、不良品を良品に見せかける |
| R    | 顧客が「評価結果を受け取っていない」と主張し、後のトラブルの原因になる | アクセスログおよびデータ閲覧履歴の記録                       |
| I    | 通信の傍受により、顧客の個人情報や製造ノウハウが含まれる画像が盗まれる | HTTPS通信、ブラウザキャッシュの制御                          |
| D    | 大量のアクセスを送り、フロントエンドをダウンさせて工場の出荷判定を止める | レートリミットの設定                                         |
| E    |                                                              |                                                              |

#### TB2

| TB2  | 緩和策                                                       | 脆弱性                                                       |
| ---- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| S    | 内部コンポーネント間の相互認証を実施する                     | 不正なプロセスが評価プロセス（Evaluate  Product Image）になりすまし、偽の評価結果を送る |
| T    | 通信の暗号化を実施する                                       | AIが判定した「Evaluation  result」を、管理コンポーネントに届く前に書き換える |
| R    | 評価プロセスの実行ログを集中管理                             | AIの評価プロセスが実行された形跡が残っておらず、誤判定の責任追跡ができない |
| I    | 内部ネットワークを流れる未暗号化の「Customer  data」が傍受される | 内部通信の暗号化、不要なデータの内部転送抑制                 |
| D    | 評価プロセスに対して過大なリクエストを送り、リソースを枯渇させる | API  Gatewayによるスロットリング、キュー処理の導入           |
| E    |                                                              |                                                              |

#### TB3

| TB3  | 緩和策                                                       | 脆弱性                                                       |
| ---- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| S    | ストレージへのアクセス制御                                   | 不正なスクリプトが「Product  Image I/O」になりすまして、ストレージ内の画像を削除・上書きする |
| T    | ファイルのハッシュ値管理、不変ストレージ（WORM）の検討       | ストレージ内の「Product  Images」を書き換え、AIの再学習や再評価を不具合を起こさせる |
| R    | ストレージ操作の監査ログの有効化                             | 誰がいつ画像をアップロード・変更したかの記録がなく、不正操作を特定できない |
| I    | 公開アクセスのブロック設定、暗号化（サーバー側暗号化）       | 設定ミスにより「Product  Images」フォルダが外部に公開され、製造ラインの様子が漏洩する |
| D    | ストレージクォータの設定、監視・アラート通知                 | ストレージ容量を意図的に枯渇させ、ラインカメラからの画像保存を妨害する |
| E    | I/Oプロセスの脆弱性を利用し、同一サーバー内の他のデータベースファイルへアクセスする | 権限の最小化                                                 |

## エージェント型AIを用いたアプリケーション
### STRIDEテーブル
#### TB1

| TB1                                      | 緩和策                                                      | 脆弱性                                                       |
| ---------------------------------------- | ----------------------------------------------------------- | ------------------------------------------------------------ |
| S:  Spoofing（なりすまし）               | Customer  Managementによる多要素認証（MFA）とセッション管理 | 不十分なパスワードポリシーやセッション管理によるアカウント乗っ取り（なりすまし） |
| T:  Tampering（改ざん）                  | TLSによるフロントエンドとバックエンド間の通信暗号化         | 通信経路上の暗号化不備による、リクエストデータ（旅行の好み等）の改ざん |
| R:  Repudiation（否認）                  | API  Gateway等でのアクセスログ・操作ログの取得              | ログの欠落や改ざん耐性の欠如により、ユーザーの操作を追跡できない |
| I: Information  Disclosure（情報漏えい） | セキュアな通信経路の確保、画面上の機密情報マスキング        | 通信の盗聴、または不適切なエラーメッセージによる内部システム情報の漏えい |
| D: Denial of  Service（サービス拒否）    | API  Gatewayでのレート制限、WAFの導入                       | 大量のリクエストによるシステムリソース枯渇（フロントエンドのサービス拒否） |
| E: Elevation  of Privilege（権限昇格）   | Customer  Managementのロールベースのアクセス制御(RBAC)      | 認可制御の不備による、他ユーザーの予約情報・決済情報への不正アクセス（BOLA/IDOR） |

#### TB2

| TB2                                      | 緩和策                                               | 脆弱性                                                       |
| ---------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------------ |
| S:  Spoofing（なりすまし）               | エージェント間の相互認証（mTLS等）の導入             | エージェント間の認証が不十分で、悪意あるプロセスが正規エージェントになりすます |
| T:  Tampering（改ざん）                  | エージェント間通信のデータ署名と整合性チェック       | エージェント間のメッセージ（予約データや指示）が改ざんされる |
| R:  Repudiation（否認）                  | エージェント間のリクエスト・レスポンスの監査ログ記録 | エージェント間の通信ログが不足しており、不正な指示の出所が特定できない |
| I: Information  Disclosure（情報漏えい） | メモリ内の機密情報の保護、エージェント間通信の暗号化 | 共有メモリや暗号化されていない内部通信からのPayment  dataの漏えい |
| D: Denial of  Service（サービス拒否）    | エージェントのタスクキューイングやタイムアウト設定   | Travel  Planning Agentから他のAgentへの過剰なタスク送信による内部リソース枯渇 |
| E: Elevation  of Privilege（権限昇格）   | 特化型エージェント（Payment等）の最小権限の徹底      | Travel  Planning Agentの指示だけで、Payment Agentが未認可の決済処理を実行できてしまう |

#### TB3

| TB3                                      | 緩和策                                                 | 脆弱性                                                       |
| ---------------------------------------- | ------------------------------------------------------ | ------------------------------------------------------------ |
| S:  Spoofing（なりすまし）               | シークレットマネージャーを用いたAPIキーの安全な管理    | ソースコード等にハードコードされたAPIキーが漏えいし、外部APIを不正利用される |
| T:  Tampering（改ざん）                  | 外部APIからのレスポンスのサニタイズと検証              | 外部APIの不正なレスポンスを検証せずに受け入れ、システム内部のデータが汚染される |
| R:  Repudiation（否認）                  | 外部APIの呼び出し履歴と応答内容のログ保存              | 外部APIの呼び出し失敗やエラーが記録されず、障害時の責任の所在が不明になる |
| I: Information  Disclosure（情報漏えい） | 外部APIに送信するデータからPII（個人情報）をマスキング | 送信するリクエスト内にPayment  dataなどの機密情報が含まれたまま外部に送信される |
| D: Denial of  Service（サービス拒否）    | 外部APIの呼び出し制限（サーキットブレーカーの導入）    | 外部APIの応答遅延や停止により、AITravelAppのプロセスがハングアップする |
| E: Elevation  of Privilege（権限昇格）   | 外部サービスごとに最小権限を持つ個別のAPIキーを使用    | 一つのAPIキーが漏えいすると、連携する全ての外部APIサービスが操作可能になる |
