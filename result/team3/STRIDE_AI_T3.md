# STRIDE+AI
## 内部にAI機能を有するアプリケーション
### STRIDEテーブル
#### TB1

| TB1  | 緩和策                                                       | 脆弱性                                                       |
| ---- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| S    | データソースの強力な認証と、データ取り込み時のデジタル署名による検証 | 不正なデータ提供者が信頼できるソースになりすまし、低品質な画像を混入させる |
| T    | データセットの統計的異常検知、データサニタイズ（クレンジング）、および「堅牢な学習」の採用 | 学習データに特定のトリガーを仕込み、「中身」が何であれ良品と判定する |
| R    | MLOpsパイプラインにおける不変の監査ログと、モデル・データのハッシュ値管理 | 誰がどのデータセットでモデルを訓練したか、変更履歴（リネージ）が追跡できない |
| I    | 保存データの暗号化、差分プライバシー（Differential  Privacy）の適用による個体特定防止 | 学習画像に含まれる機密情報や、モデルのアーキテクチャ・重みが外部に漏洩する |
| D    | リソースクォータ制限、および入力データのサイズ・形式バリデーション | 大量の無意味なデータや巨大なファイルを送り込み、学習リソースを枯渇させる |
| E    | パイプライン実行環境の分離（サンドボックス化）と最小権限の原則適用 | 学習ジョブの権限を悪用し、本番環境のデータストアへ不正アクセスする |

| R    | 評価プロセスの実行ログを集中管理<br />+<br />XAI（説明可能なAI）技術の導入、および推論時の入力と自信度スコアのログ記録 | AIの評価プロセスが実行された形跡が残っておらず、誤判定の責任追跡ができない<br />+<br />AIの誤判定時に、なぜその出力に至ったかの判断根拠（Explainability）が説明できない |
| I    | 内部ネットワークを流れる未暗号化の「Customer  data」が傍受される | 内部通信の暗号化、不要なデータの内部転送抑制                 |
| D    | 評価プロセスに対して過大なリクエストを送り、リソースを枯渇させる | API  Gatewayによるスロットリング、キュー処理の導入           |

#### TB3

| ---- | ------------------------------------------------------------ | ------------------------------------------------------------ |

## エージェント型AIを用いたアプリケーション
### STRIDEテーブル
#### TB1

| TB1                                      | 緩和策                                                     | 脆弱性                                                       |
| ---------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------ |
| S:  Spoofing（なりすまし）               | ディープフェイク等による生体認証回避の対策（生体検知機能） | ディープフェイク音声や画像を用いて、ユーザー認証を回避される |
| T:  Tampering（改ざん）                  | 入力プロンプトの厳格なサニタイズと意図検証（ガードレール） | プロンプトインジェクションにより、Travel  Planning AI Agentに悪意のある指示（不要な高額ホテルの予約等）が挿入される |
| R:  Repudiation（否認）                  | ユーザーの指示とAIの推論・操作結果の紐付けと透明性の確保   | AIの自律的な判断で発生した予約について、後から「ユーザーの指示ではない」と責任の所在が不明確になる（否認・追跡不能） |
| I: Information  Disclosure（情報漏えい） | 出力フィルタリングによるPIIの除去                          | チャットの推論結果を通じて、学習データ由来の他ユーザーの予約情報が漏えいする |
| D: Denial of  Service（サービス拒否）    | 入力プロンプトの複雑さの制限、使用リソースの監視           | 極端に複雑なプロンプトを送信し、LLMの計算リソースやAPI枠を枯渇させる攻撃（リソース過負荷  / Sponge Attacks） |
| E: Elevation  of Privilege（権限昇格）   | 安全制約の強化とユーザー権限の最小化設計                   | ジェイルブレイク（DAN等）によりモデルの安全制約をバイパスし、禁止された操作権限を取得される |

#### TB2

| TB2                                      | 緩和策                                                    | 脆弱性                                                       |
| ---------------------------------------- | --------------------------------------------------------- | ------------------------------------------------------------ |
| S:  Spoofing（なりすまし）               | エージェントアイデンティティの暗号化検証                  | 敵対的に操作されたデータによってAgent  Hijackingが発生し、不正なエージェントが権限を詐称する |
| T:  Tampering（改ざん）                  | エージェント間通信内容の検証とコンテキスト管理            | Travel  Planning Agentが汚染されたプロンプトを他エージェントに伝播させ、意図しないツール呼び出しが発生する（カスケード型ハルシネーション /  エージェント間通信の汚染） |
| R:  Repudiation（否認）                  | 全エージェントの推論チェーンと意思決定プロセスのログ化    | 複数のエージェントが相互作用する中で発生したエラーや誤予約において、どのエージェントの推論が原因か追跡できない |
| I: Information  Disclosure（情報漏えい） | エージェント間のデータ伝達の最小化とフィルタリング        | Travel  Planning AgentからLLMへの通信時など、本来共有不要なPaymentデータが他のエージェントに漏えいする（メモリポイズニング・データ漏えい） |
| D: Denial of  Service（サービス拒否）    | エージェント間の呼び出し回数・再帰ループの制限            | 目標の誤解や矛盾から、エージェント間で無限にフィードバックループが発生しシステムリソースを枯渇させる |
| E: Elevation  of Privilege（権限昇格）   | 特権的なツール（決済等）呼び出し時のHuman-in-the-Loop要件 | 自律的な意思決定により、Payment  AI Agentがユーザーの同意なしに勝手に決済を実行してしまう（目標のすり替え・権限の侵害） |

#### TB3

| TB3                                      | 緩和策                                        | 脆弱性                                                       |
| ---------------------------------------- | --------------------------------------------- | ------------------------------------------------------------ |

| T:  Tampering（改ざん）                  | 外部からの取得データ（RAG等）の信頼性検証     | 外部LLMから返された虚偽の情報（ハルシネーション）をエージェントが事実と誤認し、計画や目標が破壊される（意図破壊・目標操作） |
| R:  Repudiation（否認）                  | 外部LLMからの回答のハッシュ化および保存       | 外部LLM  APIが生成した内容が後から変更され、エージェントがなぜその判断をしたかの監査証跡が失われる |
| I: Information  Disclosure（情報漏えい） | 外部API送信前の自動マスキング（DLP機能）      | 外部の3rd  Party LLM APIへ送信するプロンプト内に、Customer dataやPayment dataを含めて送信してしまう（情報漏えい） |
| D: Denial of  Service（サービス拒否）    | 外部API呼び出しのタイムアウトおよびコスト監視 | 外部LLMに対して意図的に高コストな推論を要求する入力がなされ、単一リクエストで想定外の高コストが発生する |
| E: Elevation  of Privilege（権限昇格）   | サンドボックス化された環境でのツール実行      | 外部APIやLLMが生成した予期しないコード・スクリプトをエージェントがそのまま実行し、システム環境が乗っ取られる（予期しないRCE・コード攻撃） |
